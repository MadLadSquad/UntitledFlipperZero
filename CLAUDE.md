# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

UntitledFlipperZero (UFZ) — a partial C++ wrapper around the Flipper Zero C API (GUI service, Storage service). It is a source library: three `.hpp`/`.cpp` pairs with no build system of its own. It is consumed as a git submodule inside a FAP (Flipper Application Package) project and compiled by that project's `ufbt` build.

## Building

There is no standalone build. Compile it through a consuming app:

- This checkout typically lives as the `UFZ/` submodule of the FlipperTasks app — run `ufbt` in the parent directory (`..`) to compile.
- CI (`.github/workflows/ci.yaml`) does the same: it clones `MadLadSquad/FlipperTasks` recursively, copies `*.hpp *.cpp` over its `UFZ/` directory, and runs `ufbt`.
- There are no tests or linters. Pushing a `v*` tag triggers a release tarball (`release.yaml`).

## Architecture

Three modules, each a header/source pair:

- **`Common.hpp/.cpp`** — `Application` (owns everything), `ViewDispatcher` and `SceneManager` wrappers, the `Filesystem` service handle, and navigation macros (`NEXT_SCENE`, `FORCE_NEXT_SCENE`, `EXIT_SCENE`, `EXIT_APPLICATION`, `SEND_CUSTOM_EVENT`, `FREE_GUARD`).
- **`UI.hpp/.cpp`** — `View` and abstract `UWidget` base, plus one `UWidget` subclass per Flipper GUI module (Menu, Submenu, Popup, DialogEx, TextInput, TextBox, VariableItemList, Widget, ...).
- **`Filesystem.hpp/.cpp`** — `File` and `Directory` wrappers over the Storage API. `File` closes itself in its destructor; the templated `File::read(std::vector<T>&)` grows the buffer in chunks.

### Application lifecycle and the index invariant

`Application::run(widgets, userPointer, begin, tickPeriod)` (`Common.cpp`) drives everything:

1. Copies the widget list and harvests each widget's `enter`/`event`/`exit` callbacks into the `SceneManagerHandlers` arrays — **a widget's position in the vector is simultaneously its scene ID and its view ID**. Consuming apps rely on a scene enum matching the widget order; scene 0 is the entry scene.
2. Initializes the Filesystem, then calls the `begin` callback (before any GUI exists — this is where apps load persisted data).
3. Allocates the SceneManager, ViewDispatcher (each widget gets `alloc()`ed and registered under its index), and GUI, then enters scene 0 and blocks in `view_dispatcher_run` until `EXIT_APPLICATION`/`stop()`.
4. Cleanup is explicit: the caller invokes `application.destroy()` after `run()` returns (guarded by `bDestroyed`, as is `UWidget::destroy()`).

`run()` is **single-use per `Application` instance** — call it (or the constructor that forwards to it) exactly once. The `enter`/`event`/`exit` callback vectors are appended to, not cleared, so a second `run()` would build the `SceneManagerHandlers` from the previous run's stale callbacks. Reuse means constructing a fresh `Application`, not calling `run()` again.

The scene callbacks receive the raw Flipper context; the `Application*` is that context, and app state is reached via `Application::getUserPointer()`. `GET_WIDGET_P(app, Type, index)` and `RENDER_VIEW(app, index)` (`UI.hpp`) do widget lookup / view switching by that same index.

### The UFZ_COMPONENT macro

Most widget classes are generated by `UFZ_COMPONENT(Type, prefix)` (`UI.hpp`), which token-pastes the Flipper C function prefix to produce `alloc`/`free`/`reset`/`getWidgetView` (e.g. `text_input_alloc`, `text_input_get_view`). It only works when the C module follows the standard naming convention — `ByteInput`, `NumberInput`, `EmptyScreen`, and `Loading` deviate and implement those virtuals by hand. New widget wrappers should use the macro when possible and only add the public methods that forward to module-specific C functions.

### Views and ViewStacks

Every `UWidget` wraps its module's view in a Flipper `ViewStack` (`UWidget::allocateViewStack`). Extra `View`s can be layered on a widget via the constructor's `additionalViews` parameter; because allocation happens inside `run()`, per-view configuration must be registered beforehand with `View::setDeferredSetupCallback`, which fires right after the view is allocated and added to the stack.

## Conventions

- Everything is `noexcept`; raw Flipper handles are freed through `FREE_GUARD`, which null-checks and nulls the pointer.
- Wrapper methods are thin 1:1 forwards to the C API, `const`, and return `*this` (or the wrapper type) for chaining where the underlying call returns `void`. Match that style when extending.
- Keep changes compiling against both the `dev` and `release` firmware SDK channels (what CI/consumers build against).
